---
description: Triage GitHub issues via an autonomous label-based state machine — categorise, brief, close, block, with periodic backlog health scans.
---

Two modes:
- **Autonomous triage** — batch process unlabeled issues without human in the loop. Determines category, state, and execution labels. Posts briefs, closes resolved items, reports summary. When uncertain, applies `needs-info + hitl` with a comment.
- **Health pass** — non-interactive periodic scan. Surfaces label-invariant gaps, closes resolved blockers, reports drift. Does not triage individual issues.

Infer repo from `git remote`. Use `gh` for all GitHub operations.

Every comment must include the AI disclaimer at the top:
- Triage: `> *This was generated by AI during triage.*`
- Health pass: `> *This was generated by AI during a backlog health pass.*`

## Label System

### Category — exactly one
| Label | Description |
|---|---|
| `bug` | Something is broken |
| `feature` | New capability |
| `chore` | Maintenance, refactor, deps |
| `documentation` | Docs-only changes |

### State — exactly one
| Label | Description |
|---|---|
| `needs-info` | Waiting on reporter |
| `planned` | Fully specified, in backlog |
| `in-progress` | Being worked on |
| `in-review` | PR open |
| `wontfix` | Rejected, to be closed |

### Execution — required when `planned`
| Label | Description |
|---|---|
| `afk` | Agent can ship autonomously |
| `hitl` | Human-in-the-loop required |

**Invariant:** `planned` MUST have exactly one of `afk` or `hitl`.

### Priority — optional
`p1` Critical · `p2` Important · `p3` Nice to have

### Meta — optional
`blocked` · `stale` · `prd`

## Definition of Ready (DoR)

An issue must meet all criteria before receiving `afk`. Failures become `hitl`.

| # | Criterion |
|---|---|
| 1 | Problem statement — clear *why* |
| 2 | Acceptance criteria — specific, testable outcomes |
| 3 | Technical context — enough to know where to start |
| 4 | Testing strategy — which layers, mocks needed |
| 5 | Out of scope — what is NOT being done |
| 6 | Dependencies — other issues, PRs, external changes |
| 7 | No open questions — design forks resolved |

Per-category additions: `bug` needs steps to reproduce, expected vs actual, environment. `feature` needs user story and migration path. `chore` needs why-now, risk, and verification plan.

**Can a developer who has never seen this codebase build and verify from this issue alone?** If no → `hitl`.

## State Machine

| From | To | By |
|---|---|---|
| `unlabeled` | `planned + afk/hitl` | Triage — meets DoR |
| `unlabeled` | `needs-info + hitl + category` | Triage — underspecified |
| `unlabeled` | closed (`wontfix`) | Triage — spam/dup/out-of-scope |
| `needs-info` | `planned` | Future pass — reporter replied |

## Workflow: Pre-flight Backlog Scan

Run these four gap queries before triage. Fix invariants in bulk.

**Gap A** — `planned` without `afk`/`hitl`:
```bash
gh issue list --label planned --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select((.labels | map(.name) | contains(["afk"]) | not) and (.labels | map(.name) | contains(["hitl"]) | not)) | "#\(.number) \(.title)"'
```

**Gap B** — no state label:
```bash
gh issue list --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select(((.labels | map(.name)) | (contains(["planned"]) or contains(["needs-info"]) or contains(["in-progress"]) or contains(["in-review"])) | not)) | "#\(.number) \(.title)"'
```

**Gap C** — PRD-titled missing `prd` or `hitl`:
```bash
gh issue list --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select(.title | test("^(feat\\(PRD\\)|PRD:)")) | select((.labels | map(.name) | contains(["prd"]) | not) or (.labels | map(.name) | contains(["hitl"]) | not)) | "#\(.number) \(.title)"'
```

**Gap D** — `blocked + planned`:
```bash
gh issue list --label blocked --label planned --state open --limit 200 --json number,title \
  | jq -r '.[] | "#\(.number) \(.title)"'
```

## Workflow: Autonomous Triage

### Step 1: Gather context per issue
- Read body, comments, labels, reporter, timestamps
- Parse prior triage notes; do not re-ask resolved questions
- Explore codebase for domain context and relevant interfaces
- Check for duplicates: `gh issue list --search "<keyword>"`
- Verify referenced files/functions exist. Stale → close + refile.
- Check if the feature already shipped.

### Step 2: Evaluate DoR

| DoR result | Action |
|---|---|
| All criteria met | `planned + afk` — post brief |
| Criteria not met, otherwise clear | `planned + hitl` — post DoR gap comment |
| Uncertain / insufficient context | `needs-info + hitl + category` — post triage notes |
| Spam / duplicate / out-of-scope | Close as `wontfix` |
| Blocked | Apply `blocked`, remove `planned`, document unblock condition |
| Already shipped | Close with evidence |

Self-check before posting: `planned` MUST have `afk` or `hitl`.

### Step 3: Post-run summary
```markdown
> *This was generated by AI during triage.*
## Triage Summary — <ISO date>
**Processed:** <N>
| Issue | Outcome |
|---|---|
| #N | planned + afk |
```

## Workflow: Health Pass

Non-interactive. Read-only except Pass 1. Do not use for batch triage.

- **Pass 0** — run Gap A–D queries. Surface counts and issue numbers. Do not auto-relabel.
- **Pass 1** — close stale blockers: for each `blocked` issue, check if all referenced blockers are closed. If yes, remove `blocked` and post disclaimer comment.
- **Pass 2** — in-progress drift: `in-progress` + `updatedAt` > 14d + no open PR → flag.
- **Pass 3** — stale planned: `planned + afk` + `createdAt` > 60d → surface.
- **Pass 4** — priority drift: `p1` older than 7d → surface.
- **Pass 5** — report all findings.

## Constraints

### MUST DO
- Include the AI disclaimer in every comment.
- Verify every file path and function reference exists before posting an AFK brief.
- Apply the DoR before assigning `afk`.

### MUST NOT DO
- Auto-label `afk` when DoR is not met.
- Remove or change labels without posting a comment explaining why.
- Assign `planned` without exactly one of `afk` or `hitl`.
