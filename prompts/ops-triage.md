# GitHub Issue Triage

Two modes:
- **Autonomous triage** — batch process unlabeled issues without human in the loop. Determines category, state, execution labels. Posts briefs, closes resolved items, reports summary. When uncertain, applies `needs-info + hitl` with a comment.
- **Health pass** — non-interactive periodic scan. Surfaces label-invariant gaps, closes resolved blockers, reports drift. Does not triage individual issues.

Infer repo from `git remote`. Use `gh` for all GitHub operations.

## AI Disclaimer

Every comment must include the disclaimer at the top. Triage: `> *This was generated by AI during triage.*` Health pass: `> *This was generated by AI during a backlog health pass.*`

## Reference: Definition of Ready

Issue must meet criteria before `afk`. Failures become `hitl`.

| # | Criterion | Check |
|---|---|---|
| 1 | **Problem statement** | Clear _why_. Anyone understands what problem is solved. |
| 2 | **Acceptance criteria** | Specific, testable outcomes. |
| 3 | **Technical context** | Enough domain context to know where to start. |
| 4 | **Testing strategy** | Which layers? Mocks needed? |
| 5 | **Out of scope** | What is NOT being done. |
| 6 | **Dependencies** | Other issues, PRs, or external changes. |
| 7 | **No open questions** | Design forks resolved. |

Per-category additions:

| Category | Additional requirements |
|---|---|
| `bug` | Steps to reproduce, expected vs actual, environment. |
| `feature` | User story, wireframe/data flow, migration path. |
| `chore` | Why now? Risk. Verification plan. |
| `documentation` | What docs, where, target audience, links. |

Before `afk`, all criteria must have substance. **Can a developer who has never seen this codebase build and verify from this issue alone?** If no → `hitl`.

## Labels

Repo-specific labels allowed but not managed here.

### Category — exactly one

| Label | Description |
|---|---|
| `bug` | Something is broken |
| `feature` | New capability |
| `chore` | Maintenance, refactor, deps |
| `documentation` | Docs-only changes |

### State — exactly one

| Label | Description |
|---|---|
| `needs-info` | Waiting on reporter |
| `planned` | Fully specified, in backlog |
| `in-progress` | Being worked on |
| `in-review` | PR open |
| `wontfix` | Rejected, to be closed |

### Execution — required when `planned`

| Label | Description |
|---|---|
| `afk` | Agent can ship autonomously |
| `hitl` | Human-in-the-loop required |

**Invariant:** `planned` MUST have exactly one of `afk` or `hitl`. When closing or blocking, remove `planned`.

### Priority — optional

| Label | Description |
|---|---|
| `p1` | Critical |
| `p2` | Important |
| `p3` | Nice to have |

### Meta — optional, coexists with any state

| Label | Description |
|---|---|
| `blocked` | Waiting on dependency |
| `stale` | No recent activity |
| `prd` | Has associated PRD |

**Conflict rule:** more than one state label → flag in summary, do not resolve.

**Non-planned execution labels:** `hitl` can coexist with `needs-info`. Signals future need. Do not remove it.

## State Machine

| From | To | By |
|---|---|---|
| `unlabeled` | `planned` + `afk`/`hitl` | Triage — meets DoR |
| `unlabeled` | `needs-info` + `hitl` + category | Triage — underspecified or uncertain |
| `unlabeled` | closed (`wontfix`) | Triage — spam/dup/out-of-scope |
| `needs-info` | `planned` | Future pass — reporter replied |
| `planned` | `in-progress` | `/ship` |
| `in-progress` | `in-review` | `/ship` |
| `in-review` | closed | `/ship` |

`blocked` and `stale` are overlays — added/removed without changing state.

## Reference: Label Gap Queries

Used by pre-flight scan and health pass (Pass 0). Each returns one line per issue (`#N title`). Empty = clean.

### Gap A — `planned` without `afk`/`hitl`

```bash
gh issue list --label planned --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select((.labels | map(.name) | contains(["afk"]) | not) and (.labels | map(.name) | contains(["hitl"]) | not)) | "#\(.number) \(.title)"'
```
### Gap B — no state label

```bash
gh issue list --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select(((.labels | map(.name)) | (contains(["planned"]) or contains(["needs-info"]) or contains(["in-progress"]) or contains(["in-review"])) | not)) | "#\(.number) \(.title)"'
```
### Gap C — PRD-titled missing `prd` or `hitl`

```bash
gh issue list --state open --limit 200 --json number,title,labels \
  | jq -r '.[] | select(.title | test("^(feat\\(PRD\\)|PRD:)")) | select((.labels | map(.name) | contains(["prd"]) | not) or (.labels | map(.name) | contains(["hitl"]) | not)) | "#\(.number) \(.title)"'
```
### Gap D — `blocked` + `planned`

```bash
gh issue list --label blocked --label planned --state open --limit 200 --json number,title \
  | jq -r '.[] | "#\(.number) \(.title)"'
```
### Patterns

- `feat(PRD):` / `PRD:` → `prd + hitl + planned`
- `Deepen` / `Lift` / `Fix` → architecture refactors; usually `afk` once brief has context
- `bug:` → from `plan-bug`; should be fully labelled
- Old `[AFK]`-suffixed → predate labelling; audit before closing

## Workflow: Pre-flight Backlog Scan

Run the four gap queries. If non-empty, fix invariants in bulk using patterns above.

## Workflow: Autonomous Triage

Triggered by "triage all unlabeled issues", "process the backlog", or any batch directive. No human interaction.

### Step 1: Gather context

For each issue:
- Read body, comments, labels, reporter, timestamps
- Parse prior triage notes; do not re-ask resolved questions
- Explore codebase for domain context, relevant interfaces, existing behavior
- Check `.out-of-scope/*.md` for prior rejections
- Verify referenced files/types/functions exist and are live. Stale → close + refile
- Check shipped-already: does the feature already exist?
- Check duplicates: `gh issue list --search "<keyword>"`

### Step 2: Category-specific pre-checks

**Research template (non-code projects):** regulations → local providers + pricing → hardware comparison. Post as comment.

**Bug reproduction:** Attempt to reproduce. Report findings. If fails or lacks detail → `needs-info`.

### Step 3: Evaluate DoR

**Security bypass:** Issues from audits with file paths, line numbers, impact, fix recommendations → straight to `afk`.

| DoR result | Action |
|---|---|
| All criteria met | `planned + afk` — post brief |
| Criteria not met, otherwise clear | `planned + hitl` — post DoR gap comment |
| Uncertain / insufficient context | `needs-info + hitl` + category — post triage notes |

Never auto-label `afk` if DoR not met.

**DoR gap comment:**

```markdown
> *This was generated by AI during triage.*
## Triage — DoR Assessment
**Category:** <...> **State:** planned + hitl
**What is here:** <answered criteria>
**What is missing:**
- [ ] Criterion — gap
```

### Step 4: Apply outcome

| Outcome | Action |
|---|---|
| `planned + afk` | Post brief, apply labels |
| `planned + hitl` | Post DoR gap comment, apply labels |
| `needs-info + hitl` + category | Post triage notes with questions, apply labels |
| `wontfix` | Explain, close (write `.out-of-scope/` for features) |
| `blocked` | Apply `blocked`, remove `planned`, document unblock. Split actionable subset as new AFK issue. |
| `shipped-already` | Close with evidence, remove `planned` |
| `superseded` / `duplicate` | Close, link to canonical, remove `planned` |
| `stale-brief` | Close, list gaps for fresh re-filing |

Self-check: `planned` MUST have `afk` or `hitl`.

### Post-run summary

```markdown
> *This was generated by AI during triage.*
## Triage Summary — <ISO date>
**Processed:** <N>
| Issue | Outcome |
|---|---|
| #N1 | planned + afk |
```
If all clean, one line suffices.

### Needs Info output

```markdown
## Triage Notes
**What we established:** <summary>
**What we need (@reporter):** <actionable questions>
```

## Writing an AFK Brief

Before posting a brief with file/type/function references:
1. Every file path must exist — `ls` or `Read` each
2. Every named symbol must be live — `grep` across `src/`
3. Cross-check against other open issues for conflicts
4. Acceptance criteria must be achievable from named primitives

If wrong after publishing, post correction comment rather than silent edit.

## Workflow: Health Pass

Non-interactive, periodic scrub. Read-only except Pass 1. Do not use for batch triage.

### Pass 0 — Label invariant gaps

Run Gap A–D queries. Surface counts and issue numbers. Do not auto-relabel.

### Pass 1 — Close stale blockers

For each open `blocked` issue: parse `## Blocked by #N`, check if all blockers closed. If yes, remove `blocked` and post disclaimer comment. If partial, surface progress.

### Pass 1b — DoR violations

```bash
gh issue list --label "afk,in-progress" --state open --json number,title,body,labels,updatedAt
```
If body empty or fails core DoR → flag. Also flag `afk`-only with `updatedAt` > 14d and no triage comments.

### Pass 2 — In-progress drift

```bash
gh issue list --label "in-progress" --state open --json number,title,updatedAt,assignees
```
`updatedAt` > 14d + no open PR → flag.

### Pass 3 — Stale planned

```bash
gh issue list --label "planned" --state open --json number,title,createdAt,labels
```
`createdAt` > 60d + `afk` → surface.

### Pass 4 — Priority drift

`p1` older than 7d → surface.

### Pass 5 — Report

```markdown
> *This was generated by AI during a backlog health pass.*
## Backlog Health — <ISO date>
**Label invariant gaps:**
- `planned` without `afk`/`hitl`: <count> — #N1, …
- No state label: <count> — …
- PRD-titled missing `prd`/`hitl`: <count> — …
- `blocked` + `planned`: <count> — …
**Closed-blockers unblocked:** <count>
**DoR violations:** #N — <what>
**Drift candidates:** #N — last updated <date>
**Stale planned:** #N
**p1 lingering:** #N
```
If no items, "Backlog clean — no findings". Omit empty headings.

### Constraints

- Only auto-mutate Pass 1
- Never auto-close planned/in-progress, re-prioritise, or triage individual issues
