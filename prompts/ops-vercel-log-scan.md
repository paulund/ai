# Vercel Log Scan

## Prerequisites

- `VERCEL_API_TOKEN` env var (or `vercel` CLI as fallback)
- `gh` CLI authenticated to GitHub

## Phase 1 — Detect project and repo

1. Get `owner/repo` from `git remote get-url origin`.
2. Vercel project name typically matches the repo name (without owner).
3. If detection fails (empty, 404), stop and report error. Do not ask.

## Phase 2 — Scan for errors

Run both in parallel:

### Scan A — Runtime errors from latest production deployment

```bash
# Latest READY production deployment
vercel ls <project> --environment production --status READY -F json 2>&1 | sed '1d' | jq '.deployments[0]'

# Runtime logs (last 24h)
vercel logs <deployment-url> --since 24h 2>&1 > /tmp/vercel-runtime-logs.txt

# Filter errors/warnings
grep -iE "^(TIME.*\berror\b|TIME.*\bfatal\b|TIME.*\bwarn\b)" /tmp/vercel-runtime-logs.txt > /tmp/vercel-errors.txt 2>/dev/null
```

### Scan B — Failed deployments in last 24 hours

```bash
# List ERROR deployments
vercel ls <project> --status ERROR -F json 2>&1 | sed '1d' > /tmp/vercel-failed-deployments.json

# Filter recent
SINCE_MS=$(($(date -v-24H +%s 2>/dev/null || date -d '24 hours ago' +%s) * 1000))
jq --argjson since "$SINCE_MS" '.deployments[] | select(.createdAt > $since)' /tmp/vercel-failed-deployments.json > /tmp/vercel-failed-recent.json

# Inspect each
vercel inspect <deployment-url> 2>&1 > /tmp/vercel-deploy-<id>.txt
```

## Phase 3 — Triage each error

### Runtime errors

For each unique error:
1. **Extract:** error type, normalized message, source file path, request path/method, HTTP status.
2. **Trace to source:** `grep -rn "<substring>" src/ --include="*.ts" --include="*.tsx" -l`
3. **Read the source** to understand what's failing, expected behavior, and existing error handling.
4. **Test layer:** actions/cron → integration; pure logic → unit; route page → e2e. Check for existing test file — note if new file needed.
5. **Recent changes:** `git log --oneline -10 -- <source-file>`
6. **Existing issues:** `gh issue list --search "<error-type> in:title" --state open --limit 5`

**Classification:** `null-reference` | `type-error` | `missing-data` | `external-failure` | `logic-error` | `race-condition` | `config-error` (skip)

### Failed deployments

For each:
1. **Get details:** URL, ID, target, commit SHA, branch, PR, error message/step from `vercel inspect`.
2. **Fetch build logs:** `vercel build logs <deployment-url>` — fallback to REST API `curl https://api.vercel.com/v13/deployments/<id>/events?limit=100 --header "Authorization: Bearer $VERCEL_API_TOKEN"`
3. **Parse for failure** — look for `error`, `Error:`, `failed`, `Exited with code`. Identify failing step (lint, typecheck, build, test).
4. **Check CI:** `gh pr view <pr-number> --json state,statusCheckRollup`
5. **Read failing source** if build/test error — `git log --oneline -5 -- <file>`
6. **Check for existing issues** for this PR or error pattern.

**Cause categories:** `build-error` | `test-failure` | `ci-infra` | `lint-error` | `deploy-config`

### Triage output format

```
TRIAGE: <error-title>
Category: <category>
Severity: <critical|high|medium|low>
Source: <file:line> or <deployment-id>
Test layer: <integration|unit|e2e>
Existing test: <path> or "none — new file needed"
Root cause: <one-sentence explanation>
Impact: <what breaks for users>
Fix hint: <suggested direction>
```

## Phase 4 — Fingerprint and deduplicate

1. **Fingerprint** — runtime: `(errorType, normalizedMessage, sourceFile)`; deployment: `(commitSha, errorStep, normalizedErrorMessage)`.
2. **Normalize:** strip hex/UUID (`/[0-9a-f-]{8,}/gi` → `<hex>`) and numeric sequences (`/\d+/g` → `<num>`).
3. Group by fingerprint, keep highest severity. Cap at 10 groups.
4. **Skip config errors** — messages about `DATABASE_URL`, `CRON_SECRET`, `SUPABASE_*`, `NEXT_PUBLIC_*`, `VERCEL_*`.

## Phase 5 — Create GitHub issues

For each group, create an issue with `bug`, `planned`, `afk` labels.

### Skip if similar title exists

## Constraints

### MUST DO
- Auto-detect project/repo — do not ask for confirmation
- Triage every error to root cause before creating an issue
- REST API first, Vercel CLI as fallback
- Normalize messages before fingerprinting
- Skip config/env-var errors; cap at 10 error groups
- Label issues `bug`, `planned`, `afk`; skip duplicates
- Include AI disclaimer + source code context (10-20 lines) + recent git commits
- Run non-interactively

### MUST NOT DO
- Create duplicate issues for the same fingerprint
- Create issues for config errors or without triage
- Modify/fix any code — only create issues
- Scan preview deployments — production only
- Ask user for confirmation before scanning
- Use `any` type in generated code or suggestions
