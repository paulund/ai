# Plan Bug

Fast-path for logging bugs the user already understands. Interview, file, done — no codebase exploration or reproduction attempt.

## Workflow

### Phase 1 — Quick context

Confirm the repo from `git remote`. If the user already gave a one-line description, skip ahead. Otherwise ask: "What's the bug?"

### Phase 2 — Interview

Ask one question at a time. Skip anything the user has already volunteered. Required fields:

- **Title** — one-line summary in imperative present (e.g. `bug: search returns stale results after filter change`).
- **Steps to reproduce** — numbered, concrete. If the user can't produce them, recommend `hitl` — don't fabricate.
- **Expected behaviour** — one or two sentences.
- **Actual behaviour** — one or two sentences. Include any error/console output.
- **Severity** — let the user describe impact, then recommend a priority:
  - `p1` — data loss, security, blocks core flow, affects all users
  - `p2` — significant feature broken, has workaround, or affects many users
  - `p3` — edge case, cosmetic, low impact

  Show reasoning and let the user override.
- **Environment** — only when relevant: browser/OS/version, account type, feature flags. Skip for backend-only bugs.
- **First seen / regression?** — capture if known.

### Phase 3 — Recommend execution label

- Default `afk` when: clear repro, clear expected vs actual, no judgement call on the fix.
- Recommend `hitl` when: acceptance criteria are fuzzy, fix involves product/UX trade-off, or the bug touches a sensitive area (payments, auth, data migrations).

State the recommendation and reason. Let the user override.

### Phase 4 — Confirm

Show the full drafted issue (title, body, label set) before filing. Wait for explicit "yes / file it / go ahead".

### Phase 5 — File

```bash
gh issue create \
  --title "bug: <short description>" \
  --label "bug,planned,<afk|hitl>,p<1|2|3>" \
  --body "$(cat <<'EOF'
> *This was generated by AI during bug logging.*

## Summary

<one-paragraph description>

## Steps to reproduce

1. ...
2. ...
3. ...

## Expected behaviour

<...>

## Actual behaviour

<...>

## Environment

<browser/OS/version, or "n/a — backend">

## First seen

<when it started, or "unknown">

## Acceptance criteria

- [ ] <specific, testable criterion>
- [ ] Regression test added that fails before the fix
EOF
)"
```

After filing, print the issue URL.

## Constraints

**Must:**
- Include the AI disclaimer at the top of every issue body.
- Apply exactly: `bug` + `planned` + (`afk` | `hitl`) + (`p1` | `p2` | `p3`).
- Show the full draft and wait for explicit confirmation before creating.

**Must not:**
- Explore the codebase, attempt to reproduce, or read source.
- Fabricate repro steps — recommend `hitl` instead.
- Create the issue before explicit confirmation.
